# Нефункциональные требования

## Performance

- Мгновенный отклик UI на создание долга и погашение (оптимистичное обновление стора до записи в IndexedDB).
- Старт приложения: первая интерактивность после гидрации — приемлемо быстро на слабых устройствах; загрузка курса валют не блокирует первый рендер (fire-and-forget после `hydrate()`).

## Offline-first

- Полная работоспособность без сети после первой загрузки приложения (PWA): создание/погашение долгов, просмотр истории, экспорт/импорт резервной копии — без сети.
- Единственный сетевой запрос — обновление курса валют, не чаще раза в сутки, с тихим отказом и офлайн-фолбэком на последний закэшированный курс (`ratesCache` в IndexedDB).

## Надёжность хранения

- Доменные данные (контакты, долги, операции) — в IndexedDB, не в `localStorage`.
- Версия схемы в `meta` для миграций.
- Обработка ошибок записи: `lastError` в store, откат оптимистичных обновлений (см. `addRepayment`/`createDebt`/`deleteDebt` в `debt-store.ts`).
- Ошибка обновления курса валют (`ratesFetchError`) не пересекается с `lastError` доменных операций — не должна выглядеть как сбой хранилища.

## Безопасность и приватность

- Данные о долгах локальны; без передачи на сервер.
- Единственный внешний запрос (курс валют) не содержит персональных данных — только код базовой валюты.

## Масштабируемость под backend

- `StorageAdapter` — единая точка обмена данными: позже реализация с REST/GraphQL и синхронизацией.
- Идентификаторы UUID совместимы с серверными.

## Точность денежных сумм

- Суммы округляются до 2 знаков после запятой (`roundAmount`) на каждой операции создания/погашения, чтобы избежать погрешности плавающей точки при повторных пересчётах остатка.

## Доступность и UX

- Крупные зоны нажатия, читаемые контрасты (тема уже в проекте).
- Погашение больше остатка блокируется валидацией формы, а не тихо обрезается.
